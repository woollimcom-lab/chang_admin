PDF-based expansion baseline
============================

Reference
- Field-service processor PDF provided by user.
- This note treats the PDF as the source of truth for expanding the current curtain/blind item flow into a generic work-item engine.

Current diagnosis
- The database layer is partially generalized already:
  - OrderItems cate1~cate4
  - Attributes JSON
- The real bottlenecks are still hardcoded:
  - templates/includes/item_master.html input UI
  - static/js/view_ui.js edit/restore/render flow
  - main.py /api/item/save form parsing
- Adding more categories today would only increase hardcoded branches.

Execution principles
- Do not rewrite the whole /view item system.
- Keep current curtain/blind flow working first.
- Normalize data before changing UI.
- Treat curtain/blind as profile pack #1, not as the core engine.
- Add new domains only after save/edit/restore are normalized.

Canonical payload v1
--------------------

Purpose
- Every future domain-specific UI must serialize into one payload before save.
- The backend save service should depend on this payload, not on raw form branches.

Shape
```json
{
  "domain": "curtain",
  "workType": "install",
  "schemaCode": "curtain_pair_v1",
  "catalogItemId": null,
  "itemName": "암막커튼",
  "supplierId": null,
  "supplierName": "예시거래처",
  "qty": 2,
  "unit": "폭",
  "costPrice": 0,
  "sellPrice": 120000,
  "lineAmount": 240000,
  "location": "거실",
  "attributes": {
    "subType": "겉지",
    "color": "아이보리",
    "option": "나비주름",
    "width": 320,
    "height": 230,
    "memo": ""
  },
  "source": {
    "mode": "legacy-form"
  }
}
```

Minimum mapping rules
- domain:
  - curtain -> curtain
  - blind -> blind
  - current generic and future trades -> generic or their own domain code
- itemName:
  - prioritize cate1/ProductName-equivalent
- qty:
  - current Quantity
- unit:
  - curtain default 폭
  - blind default ㎡ or current business rule output
  - others profile-defined
- lineAmount:
  - current LineTotal
- attributes:
  - variable trade-specific fields only

Sprint 1 implementation boundary
--------------------------------

Goal
- Keep the current UI.
- Add a normalization layer before save.
- Do not replace the renderer yet.

Files expected in Sprint 1
- main.py
- static/js/view_ui.js
- templates/includes/item_master.html

Sprint 1 tasks
- Define canonical serializer on frontend.
- Add legacy adapter for current curtain/blind/generic form state.
- Add backend helper that converts canonical payload into OrderItem fields.
- Keep /api/item/save endpoint path unchanged.
- Keep current screen UX unchanged.

User verification checklist
---------------------------

Because auto-upload is enabled, the user must verify all of the following after each patch:

1. Deployment reflection
- Confirm the saved local file is actually uploaded to the server.
- Confirm Python file changes are reflected by reload or service restart.

2. Browser cache
- Hard refresh before testing /view.
- Especially verify static/js/view_ui.js is the latest file.

3. Core regression
- Curtain item add works
- Blind item add works
- Generic item add works
- Existing saved curtain edit works
- Existing saved blind edit works
- Existing saved generic edit works
- Item delete works

4. Mismatch symptoms to watch
- HTML changed but JS still old
- JS changed but backend still old
- Server reload did not happen after Python change
- Edit modal opens with empty or shifted values

Rollback point
--------------
- If any save/edit regression appears in /view after Sprint 1, rollback should happen at the adapter layer only.
- Do not mix renderer replacement with save normalization in the same patch.
